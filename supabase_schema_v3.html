Enums (check constraints in Postgres)
profiles.roleadmin — loads content, manages courses/lessons/quizzes. coordinator — creates classes, tracks student progress, needs subscription. independent — self-paced learner, no class, enrols directly on a course. student — joins a class via join_code, progress tracked by coordinator.
subscriptions.plancoordinator — R199/mo, 20 student slots. enterprise — custom pricing, unlimited coordinators + students, white-label.
class_members.rolecoordinator — can view progress of all students in the class. student — can only see their own progress.
course_enrolments .enrolment_typeindependent — user enrolled themselves directly on the course. class_based — user was enrolled via a class join. Both types share the same lesson_progress table.
questions.typemcq — multiple choice, one correct option. true_false — always exactly 2 options in question_options. short_answer — free text, no options, answer stored in attempt_answers.text_answer.
Role flows — how each user type moves through the system
Admin
1Signs up → profile created with role = admin
2Creates courses, adds lessons with content, attaches a quiz to each lesson
3Sets is_published = true → course becomes available to all roles
Coordinator
1Signs up → profile role = coordinator, subscription row created (R199/mo, 20 slots)
2Creates a class linked to a published course → join_code auto-generated
3Shares join_code with up to 20 students (slot count enforced by subscription.student_slots)
4Monitors lesson_progress and quiz_attempts for all students in their classes
5Can manage multiple classes across different courses simultaneously
Student (class-based)
1Signs up → profile role = student
2Enters join_code → row added to class_members + course_enrolments (type = class_based)
3Works through lessons in order → lesson_progress upserted per lesson
4Completes quiz after each lesson → quiz_attempt + attempt_answers recorded
5All lessons done + all quizzes passed → badge awarded in student_badges
6Can be in multiple classes (e.g. two different courses with two different coordinators)
Independent learner
1Signs up → profile role = independent
2Browses published courses → enrols directly via course_enrolments (type = independent)
3No class, no coordinator, no join_code needed
4Self-paced: works through lessons and quizzes at their own speed
5Same lesson_progress and quiz_attempts tables as students — no separate tracking needed
6Badge awarded on completion, same as student path
Key design decisions & scalability notes
auth.users is never duplicated. Email, phone, and created_at live there only. All tables FK to auth.users.id.
profiles.role is the single gate for RLS policies — one check per request, no joins needed to determine what a user can see.
course_enrolments is the universal enrolment table. Querying "what courses is this user in" is always one table regardless of role — filter by user_id.
lesson_progress uses an upsert pattern (INSERT ... ON CONFLICT DO UPDATE). Never insert a duplicate row — always upsert on (user_id, lesson_id).
class_members.role is separate from profiles.role — a coordinator is always a coordinator (profile), but inside a specific class they can also be listed as a student if they want to take the course themselves.
attempt_answers.option_id is nullable to support short_answer questions without a separate table.
subscription.student_slots enforced at the application layer (or a Postgres trigger) when a new student joins via join_code — count class_members where class.created_by = coordinator and compare to slots.
All timestamps use timestamptz (timezone-aware). Never use plain timestamp in Supabase — it stores without timezone and causes bugs across regions.
updated_at columns should be maintained by a Supabase trigger: moddatetime extension or a custom trigger on each table that sets updated_at = now() before update.
Supabase Storage (not this schema) should hold lesson media files and badge icons — store only the URL in lessons.content or badges.icon_url.
